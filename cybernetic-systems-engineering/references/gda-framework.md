# GDA Framework

## 1. 为什么这个 skill 需要总体设计视角

从工程实践看：

- 初级工程师关注语法和框架
- 中级工程师关注算法和模式
- 高级工程师真正解决的是：**如何控制复杂性**

这个 skill 的理论底座来自两个方向：

1. **系统工程**
   - 关注总体目标、边界、分解、集成、组织
2. **工程控制论**
   - 关注反馈、稳定性、时滞、噪声、鲁棒性、调节策略

把它们映射到软件工程，核心不是数学推导，而是：

- 把软件系统当作可观测、可控制、会受扰动、会振荡的复杂系统

## 2. 五大维度

### 2.1 第一性原理

先问：

- 业务真正目标是什么
- 物理极限是什么
- 什么是不变量
- 什么是绝对不能妥协的约束

软件中的等价问题：

- 吞吐 vs 一致性
- 延迟预算
- 网络 RTT
- 磁盘 IOPS
- 内存 / 缓存边界

对应方法：

- 约束驱动设计

### 2.2 公理化思维

系统必须建立在少数不可推翻的契约上：

- API 契约
- schema 契约
- 状态机合法跃迁
- 事件语义

软件中的反面典型：

- 模块越权访问
- 影子实现
- 同一状态有两套解释

对应方法：

- 定义即防御
- 契约优先于实现

### 2.3 多模型思维

复杂系统不能只用一种图来理解。

至少应切换三种模型：

1. 数据流模型
2. 状态机模型
3. 时序 / 并发 / 排队模型

软件中的典型价值：

- 同步 RPC 看不清的问题，换成事件流模型会更清楚
- OOP 卡住的高并发状态流转，换成 Actor / 队列模型更稳定
- 代码 profiler 看不清的延迟问题，用排队模型更直观

对应方法：

- 正交投影建模法

### 2.4 类比迁移能力

软件系统没有物理形体，但可以寻找物理同构：

- 限流 / 降级 → 漏桶 / 令牌桶
- 自动扩缩容 → PID
- 消息队列 → 水库 / 缓冲池
- 熔断 → 自动断路器

当系统出现：

- 拥堵
- 雪崩
- 震荡
- 超调

就应该主动寻找物理世界中的成熟控制经验。

### 2.5 反馈与验证能力

没有反馈回路的系统，等价于盲飞。

必须有：

- metrics
- logs
- traces
- replay / repro
- tests / benchmarks / gate

尤其要关注：

- 时滞
- 非线性
- 噪声
- 偶然通过的测试

## 3. GDA 四步法

### Step 1: Axiom & Boundary

执行动作：

1. 明确系统最终目标
2. 写出不变量
3. 列出硬约束
4. 算清物理边界

产物：

- 控制合同（Control Contract）
- 边界清单
- 风险清单

### Step 2: Multi-model Construction

执行动作：

1. 建立静态契约域
   - API
   - schema
   - 配置
   - 事件格式
2. 建立动态状态域
   - 状态机
   - 生命周期
   - 状态收口
3. 建立容量与排队域
   - 队列
   - 缓冲
   - 背压
   - 瓶颈

产物：

- 最小系统图
- 状态机图或文字版
- 容量/时滞估计

### Step 3: Cybernetic Control

执行动作：

1. 识别主要误差
2. 选择最小控制输入
3. 一次只改一类问题
4. 防止振荡

典型控制动作：

- 缩小改动范围
- 加强可观测点
- 从大修退回最小修复
- 从“改实现”切换到“补测试/补契约/修接口”

### Step 4: Closed-loop Observability

执行动作：

1. 分层验证
   - L0：快回路
   - L1：中回路
   - L2：慢回路
2. 明确什么只能在真实环境验证
3. 沉淀 review / gate / handoff 文档

产物：

- 测试矩阵
- 审计报告
- gate 文档
- handoff

## 4. 对软件架构师的直接启示

### 4.1 不是先选技术，而是先选控制目标

错误问题：

- 用什么数据库？
- 上不上微服务？

正确问题：

- 哪个指标必须优先满足？
- 哪个不变量绝不能破？
- 哪个失败模式必须被优先吸收？

### 4.2 不是先加抽象，而是先界定契约

错误问题：

- 如何把实现写得更“通用”？

正确问题：

- 什么应该是 public？
- 什么只是测试方便，不该变成公共 API？
- 什么是模块的业务区位？

### 4.3 不是所有“通过测试”都等于问题解决

必须区分：

- 语义测试
- schema 契约测试
- 主链测试
- 真实环境 gate

这是本 skill 最关键的现实原则之一。

## 5. 使用这个 skill 时的输出风格

默认要求输出这些内容中的大部分：

1. 当前误差是什么
2. 当前系统状态估计是什么
3. 当前改动为什么是最小控制输入
4. 验证层次是什么
5. 哪些风险还没被闭环吸收

## 6. 常见误区

1. 把控制论理解成“堆术语”
2. 把系统工程理解成“画更多图”
3. 把离线通过等价为真实环境通过
4. 把测试便利性凌驾于模块职责之上
5. 把临时修复变成新的影子事实源
